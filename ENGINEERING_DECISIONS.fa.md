# تصمیم‌های مهندسی BankFlow

[English](ENGINEERING_DECISIONS.md) | [فارسی](ENGINEERING_DECISIONS.fa.md)

## ۱. معماری لایه‌ای Laravel

**گزینه‌های بررسی‌شده:** Controllerهای دارای منطق داخلی، Laravel لایه‌ای با Actionها، یا معماری کامل Clean/Hexagonal با Repository و Port.

**تصمیم:** استفاده از Controllerهای کم‌حجم، Actionهای تک‌منظوره، ماژول دامنه تسهیلات و مدل‌های Eloquent. این ساختار Use Caseها و مرز Transactionها را بدون افزودن Abstractionهای غیرضروری صریح نگه می‌دارد. برای توسعه‌دهندگان Laravel آشنا و به‌سادگی قابل تست است. محدودیت آن، وابستگی بیشتر به Laravel و Eloquent در مقایسه با معماری Ports and Adapters است.

## ۲. Pipeline مرتب Stageها به‌جای انشعاب Hard-codeشده

**گزینه‌های بررسی‌شده:** شرط‌های تو در تو در یک Service، Rule Engine عمومی، یا Handlerهای مرتب و بدون State.

**تصمیم:** مدل‌سازی Workflow به‌صورت Stepهای مرتب و نسخه‌بندی‌شده پایگاه داده که از طریق Registry امن PHP Resolve می‌شوند. هر Handler نتیجه‌ای Type-safe برمی‌گرداند و Engine مسئول حرکت بین مراحل است. افزودن Stage موجب تغییر Enum، Handler، Registry، اعتبارسنجی قوانین، تعریف Stage یا نسخه Workflow و تست‌ها می‌شود؛ Engine و Handlerهای موجود بدون تغییر باقی می‌مانند.

این طراحی بدون پذیرش ورودی اجرایی مدیر، توسعه‌پذیر و قابل ممیزی است. در مقایسه با Graph یا زبان قوانین عمومی انعطاف کمتری دارد: Engine فعلی یک Pipeline خطی و مرتب با Skip مشروط و نتایج نهایی را پشتیبانی می‌کند، نه Branch یا Cycle دلخواه.

## ۳. مالکیت قوانین توسط پایگاه داده و مالکیت Handlerها توسط کد

**گزینه‌های بررسی‌شده:** همه قوانین در فایل‌های پیکربندی PHP، همه رفتار در پایگاه داده، یا رویکرد ترکیبی.

**تصمیم:** ترتیب Workflow و مقادیر کسب‌وکار به‌شکل JSON اعتبارسنجی‌شده در Stepهای نسخه‌بندی‌شده پایگاه داده ذخیره می‌شوند و `config/workflow-stages.php` کدهای امن را به کلاس‌های پیاده‌سازی نگاشت می‌کند. مدیران می‌توانند آستانه‌ها و ترتیب را بدون Deploy کد تغییر دهند، اما امکان تزریق PHP، SQL یا Expression ندارند.

مزیت این روش، انعطاف عملیاتی همراه با مرز امنیتی کنترل‌شده است. هزینه آن، نگهداری اعتبارسنجی اختصاصی هر Stage و چرخه انتشار مدیریتی است. با اینکه متن چالش خواندن قوانین از Configuration را مطرح می‌کند، پایگاه داده عمداً منبع نسخه‌بندی‌شده پیکربندی کسب‌وکار است و فایل پیکربندی PHP فقط Handlerهای امن را ثبت می‌کند.

## ۴. تخصیص Workflow تغییرناپذیر و تاریخچه پایدار

**گزینه‌های بررسی‌شده:** Resolve کردن همیشگی آخرین Workflow، کپی کامل اطلاعات Workflow روی هر Loan، یا ارجاع به یک پیکربندی نسخه‌بندی‌شده.

**تصمیم:** هنگام ایجاد Loan، مقدار `workflow_configuration_id` مربوط به نسخه منتشرشده فعلی به آن اختصاص می‌یابد. نسخه منتشرشده تغییرناپذیر است و هر رکورد تاریخچه، Step اجراشده و Snapshot قوانین را نگه می‌دارد. این کار تصمیم‌های قدیمی را بدون تکرار کل پیکربندی برای هر Loan قابل بازتولید می‌کند. در مقابل، نسخه‌های دارای ارجاع باید حفظ شوند و حذف مخرب آن‌ها مجاز نیست.

## ۵. پردازش هم‌زمان و اتمیک

**گزینه‌های بررسی‌شده:** یک Request هم‌زمان، یک Job صف برای هر Stage، یا هماهنگ‌سازی Event-driven.

**تصمیم:** همه مراحل باقی‌مانده در یک Request و یک Transaction پایگاه داده پردازش می‌شوند. API بلافاصله وضعیت نهایی پردازش خودکار را برمی‌گرداند و خطا نمی‌تواند State ناقص باقی بگذارد. قفل ردیف، بررسی وضعیت نهایی، جست‌وجوی تاریخچه و Unique Constraint تاریخچه از اجرای تکراری یا هم‌زمان جلوگیری می‌کنند.

این راهکار ساده‌ترین انتخاب قابل اتکا برای بررسی‌های سریع و Mockشده است. محدودیت اصلی آن مدت Transaction است: Integration واقعی اعتبار، تقلب یا AML ممکن است کند یا خارج از دسترس باشد. در آینده چنین Stageهایی باید به یک State Machine قابل ادامه روی Queue، همراه با Transactionهای کوتاه، سیاست Retry صریح، Timeout و Idempotency Key منتقل شوند.

## ۶. Eloquent و PostgreSQL

**گزینه‌های بررسی‌شده:** ذخیره‌سازی درون حافظه، Abstraction مبتنی بر Repository، یا ماندگاری مستقیم Eloquent روی پایگاه داده رابطه‌ای.

**تصمیم:** استفاده از Eloquent و PostgreSQL در محیط تولید Compose. Foreign Keyهای رابطه‌ای، Composite Constraintها، قفل ردیف، ذخیره قوانین JSON و Partial Unique Index سازگاری و ماندگاری پس از Restart را فراهم می‌کنند. SQLite برای تست‌ها و Image مستقل چالش باقی می‌ماند.

مزیت، ماندگاری قوی با زیرساخت سفارشی اندک است. Trade-off این است که برخی تضمین‌های تولید، به‌ویژه Partial Unique Index و رفتارهای Concurrency، مخصوص PostgreSQL هستند و تست‌های SQLite آن‌ها را به‌طور کامل پوشش نمی‌دهند.

## سه Trade-off تعیین‌کننده

1. **سادگی در برابر Throughput:** پردازش هم‌زمان شفاف و اتمیک است، اما برای وابستگی خارجی کند مناسب نیست.
2. **پیکربندی‌پذیری در برابر سادگی Schema:** جدول‌های Workflow نسخه‌بندی‌شده تغییرات امن و ممیزی را ممکن می‌کنند، اما به اعتبارسنجی و مدیریت چرخه عمر نیاز دارند.
3. **Convention فریم‌ورک در برابر قابلیت انتقال:** Actionها و Eloquent Boilerplate را کاهش می‌دهند، اما عمداً استقلال از پیاده‌سازی ماندگاری را کنار می‌گذارند.

## محدودیت‌ها و بهبودهای بعدی

بزرگ‌ترین محدودیت معماری، Workflow خطی و هم‌زمان است. با یک هفته زمان بیشتر، اولویت‌ها شامل تست Concurrency روی PostgreSQL، تست تزریق خطا و Rollback، احراز هویت و Rate Limiting برای API، نقش‌های Authorization غنی‌تر، تولید OpenAPI، مشاهده‌پذیری Latency و خطای Stageها و طراحی Queue قابل ادامه برای بررسی‌های خارجی واقعی خواهد بود.

با این حال، معماری فعلی از رشد آینده پشتیبانی می‌کند: قرارداد پایدار Stageها بررسی‌های جدید را مستقل می‌سازد، نسخه‌های تغییرناپذیر از Loanهای موجود محافظت می‌کنند، قوانین JSON سیاست اختصاصی هر Stage را ممکن می‌سازند و Actionها نقاط اتصال شفافی برای Queue یا Adapterهای خارجی فراهم می‌کنند.
